CascadeGuard Investigation Context (Phase Audit)
Date: 2026-04-08

System architecture summary
- Environment server: FastAPI app created in server/app.py via openenv.core.env_server.create_app(CascadeEnvironment, CascadeAction, CascadeObservation).
- Runtime model: openenv.yaml defines runtime=fastapi and app=server.app:app on port 8000.
- Core environment logic: server/cascade_environment.py implements reset(), step(), state data exposure, reward shaping, and done criteria.
- Typed contracts: models.py defines CascadeAction, CascadeObservation, CascadeState with Pydantic validation.
- Client: client.py wraps EnvClient and parses reset/step/state payloads.
- Baseline inference: inference.py runs task episodes, emits [START]/[STEP]/[END], calls OpenAI-compatible API, grades outcomes.
- Task/grader layer: tasks.py defines 5 tasks and seed splits; server/graders.py maps task IDs to deterministic score functions in [0,1].
- Packaging/deploy: pyproject.toml, Dockerfile (root), server/Dockerfile, and openenv push/deploy path.

Key components and roles
- inference.py: orchestrates agent loop and environment connection strategy.
- _connect_environment(): tries HF Space URL candidates first, then local Docker image fallback.
- grade_episode import + fallback: enables local/offline grading resilience.
- validate-submission.sh: local preflight for HF ping, docker build, and openenv validate.

Failure points discovered (historical + current)
- Historical Phase-2 failure: docker run exit 125 due missing local image tag in validator context when baseline tried from_docker_image first.
- Potential auth mismatch risk: HF_TOKEN intended for HF auth while Groq LLM requires GROQ_API_KEY. Current code uses GROQ_API_KEY with HF_TOKEN fallback for compatibility.
- Potential local infra risk: local Docker daemon unavailable can break local-only checks (environment-specific, not repository logic).
- Potential scoring variance risk: if grader import fails and returns constant scores.

Performance tuning notes (post-validation)
- Objective: raise medium/hard/cyber baseline behavior while preserving API contracts and score bounds in (0.01, 0.99).
- Observed heuristic baseline before tuning (holdout means):
  - task_easy ~0.91, task_medium ~0.41, task_hard ~0.38, task_gen_blackout ~0.84, task_cyberattack ~0.38.
- First tuning pass applied:
  - Environment dynamics softened (cascade damage, weather damage, overload damage) and load drift made mean-reverting.
  - Recovery effectiveness increased (RECOVERY_HEALTH 0.9).
  - Reward shaping made less punitive for actionable wait and more positive for useful coordinate actions.
  - Task-level stress/budget tuning for medium/hard/cyber.
  - Inference task budget constants aligned with task config budgets.
- Intermediate heuristic holdout means after first pass:
  - task_easy ~0.93, task_medium ~0.85, task_hard ~0.60, task_gen_blackout ~0.91, task_cyberattack ~0.77.
- Second tuning pass (hard-task stabilization):
  - task_hard budget increased, SCADA decay reduced, storm duration shortened, and hard-task partial observability made deterministic.
  - Inference hard-task budget constant aligned.
- Current code status:
  - No syntax/type errors reported in edited files.
  - Contracts unchanged (reset/step/state shape, reward in (0,1), score in [0.01,0.99]).
